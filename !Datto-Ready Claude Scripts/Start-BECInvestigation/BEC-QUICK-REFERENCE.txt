==============================================================================
         BEC INVESTIGATION QUICK REFERENCE v4.0.2.0
                Databranch - Incident Response Workflow
==============================================================================

This quick reference matches Start-BECInvestigation.ps1 v4.0.2.0 (April 2026).

WHAT CHANGED FROM v3.0.0.0:
  * Remediation is now handled through CIPP (Compromise Remediation button),
    not by manual Azure portal clicks or PowerShell commands
  * Investigation now uses FOUR scripts instead of three - Microsoft Graph
    collection was split out to Invoke-BECGraphCollection.ps1 because the
    Graph SDK and ExchangeOnlineManagement modules conflict in the same
    PowerShell session (MSAL/WAM issue, GitHub msgraph-sdk-powershell#3576)
  * Graph collection step runs in PowerShell 7 (pwsh), all other steps in
    Windows PowerShell 5.1. Microsoft officially recommends PS7 for the
    Graph SDK
  * New detections: impossible travel, session reuse (AiTM), new MFA device,
    OAuth consent phishing, role escalation, CA policy tampering,
    MailItemsAccessed Sync/throttling, bulk file downloads
  * New outputs: Analysis\Timeline.csv and Analysis\Evidence-Manifest.csv
    (SHA-256 hashes of every artifact)
  * All CSV files with time fields now include sibling _ET columns showing
    US Eastern time alongside UTC

==============================================================================
  SETUP (ONE-TIME)
==============================================================================

Download ONLY this file:
  [ ] Start-BECInvestigation.ps1

Install required PowerShell modules (if not already present):

  Windows PowerShell 5.1 - for Exchange Online collection and analysis:
    PS> Install-Module ExchangeOnlineManagement -Scope CurrentUser -Force

  PowerShell 7 - for Microsoft Graph collection:
    If you do not have PowerShell 7, install it first:
      PS> winget install --id Microsoft.PowerShell --source winget
    Then open PowerShell 7 (search 'pwsh' in Start Menu) and run:
      pwsh> Install-Module Microsoft.Graph.Authentication -Scope CurrentUser -Force

  NOTE: The scripts will auto-install / auto-update these modules if missing,
  but pre-installing them avoids mid-investigation interruptions.

Permissions required:
  [ ] Global Administrator (or equivalent combination):
        - Exchange Administrator (for EXO cmdlets and audit log search)
        - Security Reader (for Graph sign-in logs and risk detections)
        - Application Administrator (for OAuth grants and service principals)
  [ ] Access to CIPP portal for Step 0 remediation

==============================================================================
  IMMEDIATE RESPONSE (0-5 MINUTES)
==============================================================================

STEP 0 - CIPP COMPROMISE REMEDIATION (DO THIS FIRST)

  CIPP > Identity > Administration > Users > [select victim]
       > Execute Compromise Remediation

  This single click performs all of the following:
    * Blocks user sign-in
    * Resets user password
    * Disconnects all active sessions
    * Removes all MFA methods (forces re-registration)
    * Disables all inbox rules

  DO NOT use manual PowerShell or the Entra admin portal for containment.
  The centralized CIPP workflow is now the standard.

  Items NOT covered by CIPP (review manually during/after analysis):
    * Tenant-level Transport Rules (EAC > Mail Flow > Rules)
    * OAuth consents granted to malicious apps (Entra > Enterprise Apps)
    * Conditional Access policy modifications
    * New service principals / app registrations
    * Directory role memberships

STEP 1 - INITIALIZE INVESTIGATION (< 5 seconds)   [Windows PowerShell 5.1]

  PS> .\Start-BECInvestigation.ps1 -VictimEmail "victim@company.com"

  What happens:
    * Creates timestamped investigation workspace folder
    * Generates all FOUR investigation scripts in the Scripts subfolder
    * Creates Investigation.xml config file
    * Creates Investigation-README.txt with per-investigation workflow
    * Opens investigation folder in Explorer

  Optional parameters:
    -WorkingDirectory "D:\Investigations"   (default: C:\Databranch_BEC)
    -IncidentTicket "INC-12345"
    -Technician "Your Name"
    -Scope Recent                           (default - 72 hours / 3 days)
    -Scope Standard                         (168 hours / 7 days)
    -Scope Extended                         (720 hours / 30 days)
    -Scope Maximum                          (2160 hours / 90 days)
    -LookbackHours 48                       (custom, overrides -Scope)

STEP 2 - NAVIGATE TO SCRIPTS FOLDER

  PS> cd BEC-Investigation_victim_TIMESTAMP\Scripts

==============================================================================
  DATA COLLECTION - EXCHANGE ONLINE (5-15 MINUTES)   [Windows PowerShell 5.1]
==============================================================================

PS> .\Invoke-BECDataCollection.ps1

What it does:
  * Reads victim email, paths, and lookback from Investigation.xml
  * Auto-updates ExchangeOnlineManagement to 3.7.0+ if older
  * Connects to Exchange Online (interactive browser auth)
  * Collects Exchange Online artifacts:
      - Inbox rules (with suspicious rule flagging)
      - Mailbox-level mail forwarding
      - Tenant-level transport rules (forwarding only)
      - Mailbox permissions
      - Mobile devices
      - Unified Audit Log queries (10 scoped collections):
          ExchangeItem, rule manipulation, Send ops, MailItemsAccessed,
          SharePoint/OneDrive downloads, login events, MFA changes,
          role membership changes, CA policy changes, OAuth events
      - Quick message traces (Get-MessageTraceV2, last 10 days)
      - Historical message traces (async job, full window up to 90 days)
  * Writes SHA-256 hash of every collected file to Evidence-Manifest.csv
  * Force-disconnects Exchange Online at end for clean handoff to Graph step

Lookback window overrides (use either, not both):
  PS> .\Invoke-BECDataCollection.ps1 -Scope Extended       # 30 days
  PS> .\Invoke-BECDataCollection.ps1 -Scope Maximum        # 90 days
  PS> .\Invoke-BECDataCollection.ps1 -LookbackHours 48     # 48 hours custom

Optional: Skip historical trace submission on re-runs:
  PS> .\Invoke-BECDataCollection.ps1 -SkipHistoricalTraces

If prompted about existing files (re-run scenario):
  [O] Overwrite   - replace existing file
  [D] Duplicate   - create versioned copy (_v2, _v3...)
  [S] Skip        - skip this collection type

==============================================================================
  DATA COLLECTION - MICROSOFT GRAPH (3-5 MINUTES)   [PowerShell 7 / pwsh]
==============================================================================

LEAVE YOUR PowerShell 5.1 WINDOW OPEN. Open a NEW PowerShell 7 window
(search 'pwsh' in Start Menu). Then:

pwsh> cd BEC-Investigation_victim_TIMESTAMP\Scripts
pwsh> .\Invoke-BECGraphCollection.ps1

What it does:
  * Detects PS edition at startup - warns loudly if you are in 5.1
  * Defensively disconnects any lingering Exchange Online session
  * Auto-installs Microsoft.Graph submodules if missing
  * Connects to Microsoft Graph (interactive browser auth, 8 read-only scopes)
  * Collects Entra ID / Graph artifacts:
      - Sign-in logs (IP, location, session, risk levels)
      - Risky users (Entra ID Protection)
      - Risk detections (AiTM, anomalous token, leaked creds, malicious IP)
      - Current MFA / authentication methods
      - Directory role memberships
      - Enterprise apps / service principals (flags CreatedInWindow=TRUE)
      - OAuth permission grants
      - Conditional Access policies (point-in-time snapshot)
  * Appends SHA-256 hash of every Graph CSV to Evidence-Manifest.csv

WHY POWERSHELL 7 FOR THIS STEP?
  Microsoft officially recommends PowerShell 7 for the Graph SDK. The
  Graph modules have known assembly-loading issues in Windows PowerShell
  5.1 (MSAL/WAM conflict, msgraph-sdk-powershell GitHub issue #3576).
  PS7 runs on .NET 8.0 with better assembly isolation and avoids these
  problems.

IF GRAPH AUTHENTICATION FAILS:
  Error like "GetTokenAsync does not have an implementation" or
  "Object reference not set to an instance of an object"?
  Close the PS7 window completely, open a fresh PS7 window, cd to the
  Scripts folder, and re-run. Lingering EXO assemblies in the process
  space are the most common cause.

WHEN FINISHED:
  Close the PowerShell 7 window. Return to your Windows PowerShell 5.1
  window for all remaining steps.

==============================================================================
  IMMEDIATE ANALYSIS (1-2 MINUTES)   [back to Windows PowerShell 5.1]
==============================================================================

Run immediate analysis without waiting for historical traces:

PS> .\Invoke-BECLogAnalysis.ps1 -SkipMessageTraces

What it does:
  * Reads all CSV files from RawData (both EXO and Graph collections)
  * Runs 25+ detection rules across EXO, UAL, and Graph data
  * Generates Analysis\ANALYSIS-REPORT.txt (severity-ranked findings)
  * Generates Analysis\All-Findings.csv (machine-readable findings)
  * Generates Analysis\Timeline.csv (chronological event timeline across
    all data sources, with severity tags)
  * Updates Investigation.xml with findings counts
  * Opens Analysis folder in Explorer
  * Expected time: 1-2 minutes

REVIEW IMMEDIATELY:
  1. Analysis\ANALYSIS-REPORT.txt
     -> Start here. Focus on CRITICAL and HIGH findings.

  2. Analysis\Timeline.csv
     -> Chronological view of what happened when, across all sources.

  3. Reports\SUSPICIOUS-Rules_username.csv (if present)
     -> Every forwarding/redirect/deletion/move rule flagged.

  4. RawData\MailForwarding_username.csv
     -> Verify ForwardingEnabled = False post-CIPP remediation.

CRITICAL FINDINGS - MOST HANDLED BY CIPP, BUT CONFIRM:
  [ ] CIPP Compromise Remediation executed (Step 0)
  [ ] SMTP forwarding confirmed disabled (spot-check RawData CSV)
  [ ] Malicious inbox rules confirmed deleted (CIPP handles)
  [ ] All MFA methods confirmed removed (CIPP handles)
  [ ] All sessions confirmed revoked (CIPP handles)
  [ ] Password confirmed reset (CIPP handles)

ITEMS NOT HANDLED BY CIPP - REVIEW MANUALLY IF FLAGGED:
  [ ] Tenant-level transport rules removed (EAC > Mail Flow > Rules)
  [ ] Malicious OAuth app consents revoked (Entra > Enterprise Apps)
  [ ] Malicious CA policy changes reverted (Entra > CA)
  [ ] New service principals deleted if unauthorized (Entra > App Regs)
  [ ] Directory role additions reverted if unauthorized

==============================================================================
  WAITING PERIOD (15-30 MINUTES)
==============================================================================

While waiting for historical message traces to complete:

  * Review Timeline.csv for attack chain reconstruction
  * Brief incident commander / team lead with preliminary findings
  * Update the incident ticket with the workspace path
  * Identify external contacts who may need notification (see templates
    near the end of this document)

CHECK TRACE STATUS (from PowerShell 5.1):
  PS> Connect-ExchangeOnline
  PS> Get-HistoricalSearch | Where-Object {$_.ReportTitle -like "*victim*"} |
      Format-Table ReportTitle, Status, SubmitDate

  Status values:
    Running = Still processing (wait longer)
    Done    = Ready to download (proceed to next step)

==============================================================================
  RETRIEVE MESSAGE TRACES (< 1 MINUTE)   [Windows PowerShell 5.1]
==============================================================================

PS> .\Invoke-BECMessageTraceRetrieval.ps1

What it does:
  * Reads trace job IDs from Investigation.xml (no parameters needed)
  * Checks status of both sent and received traces
  * Downloads completed traces to RawData folder
  * Adds _ET (Eastern) timestamp columns to downloaded CSVs
  * Appends each downloaded file's SHA-256 hash to Evidence-Manifest.csv
  * Updates Investigation.xml (TracesCompleted=true when both done)

Safe to re-run if traces are not ready yet - downloads what's ready and
reports on the rest.

==============================================================================
  COMPLETE ANALYSIS (2-3 MINUTES)   [Windows PowerShell 5.1]
==============================================================================

PS> .\Invoke-BECLogAnalysis.ps1

What it does:
  * Re-runs all analysis including historical message trace data
  * Detects message volume spikes and external send ratio anomalies
  * Updates ANALYSIS-REPORT.txt, All-Findings.csv, Timeline.csv with
    message trace findings merged in
  * Updates Investigation.xml CompleteAnalysisCompleted=true

==============================================================================
  FINDINGS SEVERITY REFERENCE (v4.0.2.0)
==============================================================================

*** CRITICAL - IMMEDIATE ACTION ***
  * SMTP mail forwarding enabled to external address
  * Inbox rule forwarding/redirecting to external address
  * Entra ID Protection risky user flag (confirmed compromise signal)
  * High-confidence Entra risk detections (AiTM / adversary in the middle,
    anomalous token, leaked credentials, malicious IP)
  * Impossible travel (cross-country sign-ins within impossibly short window)
  * Session ID reuse across IPs (AiTM / token theft indicator)
  * New MFA device registered during investigation window
  * Victim granted OAuth consent during window (likely OAuth phishing)
  * Admin role assignments involving victim during window
  * MailItemsAccessed Sync events (full mailbox download)
  * MailItemsAccessed logging throttled (assume worst-case full access)

*** HIGH - URGENT INVESTIGATION ***
  * Inbox rule creation or deletion during window
  * Inbox rules automatically deleting messages
  * Suspicious rule name patterns (single chars, financial keywords like
    invoice/wire/payment/bank/ach/payroll/swift/iban/routing/vendor)
  * Tenant-level transport rules with forwarding actions
  * New service principals created during window (OAuth phishing)
  * Conditional Access policy modifications during window
  * Directory/admin role memberships on victim account
  * Outbound email send burst (>30 in single hour)
  * Outbound email volume spike (>50 in single day, trace analysis)
  * SharePoint/OneDrive bulk downloads (>50 events - exfiltration)
  * Failed sign-in volume (brute force / password spray)

*** MEDIUM - INVESTIGATE FURTHER ***
  * Rules moving emails to non-standard folders
  * Delegated mailbox permissions
  * Mobile devices registered
  * High external send ratio (>70% of sent mail external)
  * OAuth consent events in tenant (non-victim)
  * MailItemsAccessed hourly summary events

*** LOW - DOCUMENT AND MONITOR ***
  * Rules automatically marking emails as read
  * UAL failed login events

==============================================================================
  EVIDENCE MANIFEST AND CHAIN OF CUSTODY
==============================================================================

Analysis\Evidence-Manifest.csv records a SHA-256 cryptographic hash of
every file collected during the investigation. This supports:
  * Chain of custody for insurance submissions or law enforcement referrals
  * Tamper detection - re-hash any file and compare to the manifest value

To verify a file:
  PS> Get-FileHash -Path <full file path> -Algorithm SHA256

Compare the returned hash to the SHA256 column in Evidence-Manifest.csv.
If they match, the file has not been altered since collection.

==============================================================================
  TIMESTAMP CONVENTIONS
==============================================================================

All CSV files with time fields have a sibling _ET column showing the same
time in US Eastern (EST or EDT, auto-handled across DST). Reports in
ANALYSIS-REPORT.txt show UTC first with Eastern in parentheses. This
supports easy correlation between Microsoft's native UTC timestamps and
the technician's local clock.

Example CSV columns after enrichment:
  CreationDate          CreationDate_ET
  2026-04-18 14:23:05   2026-04-18 10:23:05 EDT

==============================================================================
  TROUBLESHOOTING
==============================================================================

ERROR: "Investigation.xml not found"
FIX:   You are not in the Scripts folder.
       cd BEC-Investigation_victim_TIMESTAMP\Scripts

ERROR: "Failed to connect to Exchange Online"
FIX:   * Verify Global Admin or Exchange Administrator role
       * Check MFA device is available for interactive auth
       * Try: Connect-ExchangeOnline manually first to isolate
       * Confirm ExchangeOnlineManagement module version is 3.7.0+

ERROR: "Failed to connect to Microsoft Graph"
       "GetTokenAsync does not have an implementation"
       "Object reference not set to an instance of an object"
FIX:   * If in PowerShell 5.1: switch to PowerShell 7 (pwsh). The Graph
         SDK is unreliable in 5.1 due to .NET Framework assembly loading.
       * If in PowerShell 7 and it still fails: close the PS7 window
         completely (EXO assemblies may be lingering from a previous step),
         open a fresh PS7 window, cd to Scripts folder, re-run.
       * Confirm Global Admin consent for the 8 Graph scopes was granted.
       * Confirm tenant has Entra ID P2 if attempting to read risk data.
         Without P2, riskyUsers/riskDetections collections will be empty
         but not fatal - other Graph collections still work.

"Traces not ready" message
NORMAL: Historical searches take 15-30 minutes. Re-run
        Invoke-BECMessageTraceRetrieval.ps1 after waiting.

"No findings in analysis"
CHECK:  * Review Logs\DataCollection_*.log and Logs\GraphCollection_*.log
        * Confirm RawData folder contains CSV files from BOTH collection
          steps (EXO files start with UAL-/QuickTrace-/InboxRules-/etc,
          Graph files start with Graph-)
        * May be a true negative - account may not have been compromised

Module not installed:
FIX:   PS 5.1: Install-Module ExchangeOnlineManagement -Scope CurrentUser -Force
       PS 7:   Install-Module Microsoft.Graph.Authentication -Scope CurrentUser -Force

Module too old (EXO below 3.7.0):
FIX:   PS> Update-Module ExchangeOnlineManagement -Force
       If that fails:
       PS> Install-Module ExchangeOnlineManagement -Force -AllowClobber

Script will not start with "CmdletBinding can only be used inside a param
block" error:
FIX:   This is a v4.0.0.0 bug fixed in v4.0.1.0+. Upgrade the
       Start-BECInvestigation.ps1 deployment file to the current version.

==============================================================================
  COMMON ATTACK PATTERNS
==============================================================================

PATTERN 1: Phishing -> Credential Harvest -> Internal Spreading
  Day 0:   User clicks phishing link, credentials captured
  Day 0-1: Attacker reads email, creates forwarding/deletion rules
  Day 1-3: Sends phishing to contacts from victim account
  Indicators: Forwarding rules, high outbound volume spike, send burst

PATTERN 2: Invoice Fraud Setup
  Day 0:   Compromise
  Day 1-3: Attacker studies payment patterns and contacts
  Day 3-7: Creates keyword-based forwarding for invoice-related emails
  Indicators: Forwarding rules with financial keywords (invoice, wire,
              payment, ach, routing, etc.), suspicious rule name patterns

PATTERN 3: Executive Impersonation
  Day 0:   Compromise of lower-level account
  Day 1-2: Searches for executive email patterns, reads mailbox
  Day 2-3: Impersonates executive to target finance / HR
  Indicators: Unusual external recipients, abnormal send patterns,
              MailItemsAccessed Sync events

PATTERN 4: AiTM (Adversary-in-the-Middle) Token Theft
  Day 0:   User clicks MFA phishing link (evilginx, Caffeine, etc.)
  Day 0:   Attacker captures session cookie, replays from their IP
  Day 0+:  Attacker continues to use stolen session, optionally adds
           their own MFA method as persistence
  Indicators: Session ID reuse across IPs, impossible travel, risk
              detections for "adversaryInTheMiddle" or "anomalousToken",
              new MFA device registration during window

PATTERN 5: OAuth Consent Phishing
  Day 0:   Attacker sends OAuth consent request disguised as legitimate app
  Day 0:   Victim clicks Accept, granting mailbox/data scopes to attacker app
  Day 0+:  Attacker uses OAuth token (survives password reset!) to
           exfiltrate mail via Graph API
  Indicators: New service principal created in window, OAuth consent
              granted by victim, Graph API sign-ins from attacker app
