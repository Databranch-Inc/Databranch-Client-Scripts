Project Overview

ArnotOnboarding is a Windows Forms desktop application that replaces a paper/Word-document workflow at Arnot Realty Corporation. When a new employee is hired, HR staff run a guided wizard that collects all required IT onboarding information, auto-saves everything locally as JSON drafts, and on finalization generates a formatted PDF and companion JSON file written to a network share. A record library tracks all past onboardings with search and re-edit capability.

The application is designed for multi-user operation in a shared network environment where multiple HR staff may be open simultaneously. All concurrent-edit conflicts are managed via a file-locking system using sidecar .lock files on the network share.

Architecture & Components

The application uses a persistent left-rail shell (MainShell) with a single content panel that swaps views. All views are custom-painted UserControls. No Designer-generated layout code is used for content — all control positioning is done programmatically in code-behind using absolute coordinates against a consistent column grid defined in WizardPageBase.

About Button Hover — Non-Flickering Pattern

A key lesson from this project: calling _navPanel.Invalidate() unconditionally on every MouseMove event causes severe flickering because WinForms repaints the entire panel (including the logo image and all text) on every mouse-move event — which fires dozens of times per second. The correct pattern is:

// Track hover state as a field — only repaint when state CHANGES private bool _aboutBtnHovered = false; private Rectangle _aboutBtnRect; // Set during paint private void navPanel_MouseMove(object sender, MouseEventArgs e) { bool overAbout = !_aboutBtnRect.IsEmpty && _aboutBtnRect.Contains(e.Location); // Update cursor for all interactive regions _navPanel.Cursor = (overAbout || overNavItem) ? Cursors.Hand : Cursors.Default; // ONLY invalidate when hover state transitions — not on every move if (overAbout != _aboutBtnHovered) { _aboutBtnHovered = overAbout; _navPanel.Invalidate(_aboutBtnRect); // Invalidate only the 24x24 button rect } } private void navPanel_MouseLeave(object sender, EventArgs e) { if (_aboutBtnHovered) { _aboutBtnHovered = false; _navPanel.Invalidate(_aboutBtnRect); } _navPanel.Cursor = Cursors.Default; }
✅
Rule: Never Invalidate on Every MouseMove
For any custom-painted panel with hover effects, always use a stored boolean state field and only call Invalidate() when that state transitions. Where possible, pass the affected Rectangle to Invalidate() so only that region is repainted. This is especially important when the panel contains images (like the logo) which are expensive to redraw.

Auto-Save Integration

Each wizard page wires all its controls to call WizardView.OnFieldChanged() via the shared AutoSaveTimer. The debounce interval is 750ms by default (configurable in Settings). In addition, every Back/Next navigation triggers an immediate synchronous save flush so no data is lost between pages. The auto-save writes the current record state to the local draft JSON — not to the network.

File Locking System

The locking system prevents two users from editing the same finalized onboarding record simultaneously. It uses a simple JSON sidecar file (.lock) placed next to the record's network JSON. No database or server component is required.

Lock Lifecycle

EventAction
User clicks Edit on a recordLockManager.TryAcquireLock() — checks for existing .lock file; if absent, writes one with current user info

Nav Warning Banner

While any record is locked for editing by the current user, a persistent amber warning banner appears in the nav rail below the nav items. It reads: "⚠ Editing [N] completed onboarding[s] — not yet exported — [This file is/These files are] locked for other users." This banner is repainted on every nav panel paint and updates dynamically as locks are acquired or released.

LockManager Implementation Notes

The LockManager maintains a Dictionary<string, string> of recordId → lockFilePath for all currently-held locks. All file operations are wrapped in try/catch because the network path may be transiently unavailable. The IsStale property on LockFile uses a TimeSpan.FromHours(2) threshold and compares against DateTime.UtcNow.

⚠
Lock File Path Consistency
The lock file path must be derived from the same base path as the JSON it protects: jsonPath + ".lock". This means the lock file for Smith_Jane_Onboarding.json is always Smith_Jane_Onboarding.json.lock. Do not use the PDF path or the directory path as the base — only the JSON path, because the JSON filename is the stable unique identifier.

MigraDoc PDF Layout

The PDF uses MigraDoc's document object model rather than PdfSharp's low-level graphics API. This produces well-structured, reflowable content. The document uses a two-column table layout for field/value pairs, matching the visual structure of the original Arnot Word form. Section headers use a colored background row. The PDF footer shows the employee name, finalized timestamp, and app version.

Outlook Integration — Multi-Version Detection

The ExportManager detects which version of Outlook is installed by reading the Windows Registry HKCU\Software\Microsoft\Windows\Shell\Associations\UrlAssociations\mailto\UserChoice key. The ProgId value is checked: if it contains "Outlook" and matches the new Outlook identifier, an .eml file is generated and launched. If Classic Outlook is detected, COM interop via Microsoft.Office.Interop.Outlook is used to create a draft with the PDF and JSON as attachments. If neither is detected, a mailto: URL is constructed as fallback.

Network Share Re-Query (RequeryNetworkShare)

The Refresh button in RecordLibraryView triggers ExportManager.RequeryNetworkShare() which scans the configured HR path directory tree for *.json files (excluding *.lock files), reads each record's metadata, and rebuilds the RecordIndex. This allows one user's finalized records to appear in another user's Past Records list without any server component. Stale index entries (pointing to paths that no longer exist on the share) are pruned automatically during this scan.

Typography

IBM Plex Sans (the CSS design system font) is unavailable as a WinForms system font. The app uses Segoe UI as the primary font and Consolas for monospace elements. Font sizes: 16pt headings, 10pt body, 9pt labels/metadata, 8pt captions. All fonts are pre-instantiated in AppFonts and reused — never new Font() inline in control factory methods, which causes handle leaks.

Theme Implementation

ThemeHelper Strategy

Standard WinForms controls don't natively support dark theming. The approach used is: ThemeHelper.ApplyTheme(Control root) recursively walks the control tree and sets BackColor and ForeColor based on control type. After this pass, custom-painted controls (the nav rail, progress bar strip) handle all painting themselves via OnPaint overrides.

Controls that cannot be themed adequately through BackColor/ForeColor alone (DataGridView, DateTimePicker, ComboBox) are styled with specific workarounds documented inline in the relevant view files.

Custom Paint Pattern for Panels

All structural regions that require precise visual control (nav rail, wizard progress bar, modal headers) use a common pattern: the Panel's Paint event is handled and all rendering is done with Graphics primitives. The panel's BackColor is set to AppColors.SurfaceRaised to prevent white flash during resize, but all actual pixel content is drawn in the handler. Hit rectangles for interactive elements are stored as fields and updated during paint so that MouseClick and MouseMove handlers can perform accurate hit testing without needing child controls.

Custom Painting Patterns

Variable Scope in navPanel_Paint

The nav panel paint method is large and declares several local variables at different nesting levels. A recurring pitfall is CS0136: variable name shadowing — declaring int textX or float textX in an inner block when the name was already used in an outer block. The solution is to use descriptive names scoped to their context: logoTextX for the logo section, bannerTextX for the warning banner section, itemTextX for nav items.

Logo Image Caching

The nav logo image is loaded once from disk on first paint and stored in a private Image _navLogo field. It is disposed in FormClosing. Never load images inside paint handlers — disk I/O in a paint event causes visible stuttering on every repaint.

private Image _navLogo; protected override void OnPaint(PaintEventArgs e) // or navPanel_Paint { if (_navLogo == null && File.Exists("databranch_nav.png")) _navLogo = Image.FromFile("databranch_nav.png"); if (_navLogo != null) e.Graphics.DrawImage(_navLogo, logoRect); } private void MainShell_FormClosing(object sender, FormClosingEventArgs e) { _navLogo?.Dispose(); }

Bug Log & Resolutions

All significant bugs encountered during development, their root causes, and the applied fixes. This section is the primary reference for future maintainers and for templating purposes.

BUG-001
CS0136 — Variable 'textX' cannot be declared in this scope
Resolved · v1.7.0.5
Symptom
Build error: "A local or parameter named 'textX' cannot be declared in this scope because that name is used in an enclosing local scope." Occurring in MainShell.cs around the nav panel paint method.
Root Cause
The nav panel paint method declares int textX near the top for logo text positioning. A later addition (the lock warning banner) independently declared float textX inside a nested if block in the same method. C# does not allow the same name to be re-declared in any inner scope if it exists in any enclosing scope, even if the type differs.
Fix
Renamed the banner-specific variable to bannerTextX. General principle: in large paint methods, always use context-qualified variable names rather than generic ones like x, y, textX.
BUG-002
CS0219 — Variable 'y' is assigned but its value is never used
Resolved · v1.7.0.5
Symptom
Compiler warning in AboutDialog.cs line 107: "The variable 'y' is assigned but its value is never used."
Root Cause
During iterative development of BuildLayout(), the original layout approach used a running int y = 0 counter for manual positioning. When the layout was redesigned to use DockStyle and FlowLayoutPanel, the y variable was abandoned but not removed.
Fix
Removed the unused variable declaration. Lesson: when refactoring layout approaches, always scan for leftover coordinate variables.
BUG-003
CS1519 — Invalid token '{' / orphaned method body after str_replace insertion
Resolved · v1.7.0.7
Symptom
Cascade of syntax errors starting with "Invalid token '{' in a member declaration" at line 525, followed by tuple-related errors. The file appeared structurally broken.
Root Cause
A string-replace operation inserted navPanel_MouseLeave() using the opening line of ShowAboutDialog() as the anchor. The replacement text included the new method but re-used the anchor line as both the end of the new method and the start of the next. This caused the ShowAboutDialog method signature to be consumed by the replacement, leaving only its body { ... } floating at class level — which the compiler sees as a malformed member declaration.
Fix
Restored the orphaned private void ShowAboutDialog() signature. General principle: when inserting a new method adjacent to an existing one, always include the complete method signature of both the new method AND the next existing method in the replacement to prevent accidentally dropping the signature.
BUG-004
Nav panel flickering/blinking on mouse movement
Resolved · v1.7.0.6
Symptom
The entire nav rail — including text, logo, and nav items — visibly blinked and flickered whenever the mouse moved anywhere over the left navigation panel.
Root Cause
The navPanel_MouseMove handler called _navPanel.Invalidate() unconditionally on every mouse move event to keep the About button hover highlight current. Mouse move events fire at extremely high frequency (50–100+ times per second during movement), causing the entire custom-painted panel to redraw constantly. The panel contains a logo image load, text rendering, and dozens of GDI+ draw calls — all executing on every pixel of mouse movement.
Fix
Introduced private bool _aboutBtnHovered state field. MouseMove only calls Invalidate(_aboutBtnRect) (region-limited) when the hover state actually transitions (true→false or false→true). MouseLeave clears the state and invalidates the button rect. The paint method reads _aboutBtnHovered directly rather than performing a live cursor position check. Result: zero repaints during normal mouse movement; exactly one targeted repaint when the cursor enters or exits the button boundary.
BUG-005
Lock file not released after re-export from edit mode
Resolved · v1.7.0.0
Symptom
After editing and re-exporting a finalized record, the .lock file remained on the network share, permanently blocking other users from editing that record.
Root Cause
ExportManager was deriving the lock file path from the PDF filename rather than the JSON filename. When the record was originally created with one naming pattern and the re-export used a slightly different format, the paths didn't match and the lock file couldn't be located for deletion.
Fix
Lock file path is always jsonPath + ".lock" — derived solely from the JSON file path, which is the stable canonical identifier for a record. The ExportManager's lock-release step was updated to use this derivation consistently.
BUG-006
PdfSharp TypeInitializationException at startup
Resolved · v1.4.0.0
Symptom
Application crashed on first PDF export attempt with TypeInitializationException deep inside PdfSharp internals, referencing WPF imaging APIs.
Root Cause
The standard PDFsharp-MigraDoc NuGet package targets WPF imaging. In a WinForms-only project targeting .NET Framework 4.8, the WPF PresentationCore assembly is not loaded and the imaging type initializer fails.
Fix
Switched to the GDI variant: PDFsharp-MigraDoc-gdi. This package uses GDI+ throughout and has no dependency on WPF assemblies. The package reference in packages.config must explicitly reference the GDI variant by its full package ID.
BUG-007
Draft index out of sync — records appeared as "in progress" after finalization
Resolved · v1.5.3.0
Symptom
After finalizing a record, it still appeared in the DraftListView. Deleting from the draft list caused an unhandled exception because the file no longer existed.
Root Cause
The ExportManager was removing the draft file from disk but not updating the in-memory draft index object before triggering a UI refresh. The DraftListView refreshed from the stale in-memory list.
Fix
ExportManager now calls DraftManager.RemoveDraftFromIndex(recordId) immediately before triggering any UI refresh. The index file on disk and the in-memory list are kept in sync atomically.

Custom Paint Rules

When writing custom-painted panels: (1) never call Invalidate() unconditionally on every MouseMove — always gate on a state change; (2) never load images inside paint handlers — cache them as fields; (3) always use context-qualified variable names to avoid CS0136 in large paint methods; (4) store hit rectangles as fields updated during paint for use by mouse event handlers; (5) set panel BackColor to prevent white flash during resize even when painting everything manually.

Changelog

A CHANGELOG.md at the solution root tracks all version increments with date, version number, and change summary in newest-first order. The About dialog changelog section must be kept in sync manually with each build increment.
