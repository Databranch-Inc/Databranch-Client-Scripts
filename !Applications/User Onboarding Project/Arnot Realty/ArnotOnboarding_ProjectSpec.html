Project Overview

This application streamlines the process by which Arnot Realty staff submit new employee onboarding requests to Databranch. Rather than filling out a Word document and emailing it, the requestor runs a guided wizard that collects all required information, saves it to a network share as a structured PDF + JSON pair, and maintains an indexed library of all onboarding records.

The application is designed for Arnot Realty as the initial client, but its architecture must support rapid reconfiguration for other Databranch MSP clients. Customer-specific data (form fields, email domain, branding, network paths) is externalized into a configuration layer so a new customer profile can be applied without recompiling.

Non-Goals (v1.0)

The engineer-facing automation tool that consumes the JSON is explicitly out of scope for this version. The JSON schema is designed to support it in the future. No web or cloud components. No Active Directory integration in this app (that belongs in the engineer tool).

Schema Versioning

Every JSON file includes "schemaVersion": "1.0". When field changes require model updates, the version increments and the app includes a migration handler to upgrade older records on open.

Stale Lock Handling
If a lock file is older than 2 hours, the app prompts the user: "This record appears locked by [Name] on [Machine] since [time]. That lock may be stale. Would you like to override it?" Overriding deletes the lock and allows editing. This prevents permanent lock-outs if the app crashes without cleanup.

Wizard Navigation

The wizard shows a horizontal progress bar or step indicator at the top displaying all pages with the current page highlighted. Left/Right arrow buttons and Back/Next labeled buttons both navigate. The wizard remembers the last active page when resuming a draft. Clicking a step indicator dot can jump directly to any previously visited page.

WinForms Dark Mode Note
Standard WinForms controls don't fully support dark theming. A static color constants class (AppColors) will be defined at project start. Custom-painted controls (nav sidebar, progress steps) will use OnPaint overrides. Standard controls (TextBox, CheckBox, RadioButton) will have BackColor/ForeColor set in a central ThemeHelper.ApplyTheme(Control) method called on form load.

Typography

IBM Plex Sans is not available as a WinForms system font by default. The application will use Segoe UI as the primary font (closest match available on all Windows 10/11 systems) with Consolas for monospace/code elements (file paths, version numbers, usernames). Font sizes: headings 14‚Äì16pt, body 10pt, labels 9pt.

Open Questions

The following decisions need your input before or during the relevant build phase. They do not block Phase 1.

OQ-01
Network path: Do you have a specific UNC path or mapped drive letter already in use at Arnot for storing documents like this? This will inform the Settings default value and path validation logic.
OQ-02
In-progress vs. network drafts: Confirmed approach is local %AppData% for drafts. Is there any need for in-progress drafts to also sync to the network (for use on another machine), or is strictly local fine?
OQ-03
Multi-client architecture: One EXE per customer (recompile with new config) or one EXE that loads a customer profile at runtime from a JSON file? Runtime switching is more flexible; compiled-per-client allows tighter branding. Which do you prefer?
OQ-04
Requestor identity: Is the requestor always a single person (one profile), or do multiple Arnot staff share the app and need separate requestor profiles? This affects whether "My Information" is single-profile or a selectable list.
OQ-05
Application list (Page 7): Please supply the full list of applications from the original form (the Word doc text wasn't fully parseable from the upload). This will populate the customer profile config for Arnot.
OQ-06
PDF branding: Should the exported PDF include the Databranch logo, Arnot Realty logo, or both? And should it look exactly like the original Word form, or is a cleaner formatted version acceptable?
OQ-07
ILMerge vs. self-contained: For the final EXE, do you want a single EXE with all dependencies merged (ILMerge, slightly larger file) or is copying the EXE + DLLs to a shared folder acceptable? Single EXE is cleaner for distribution.

Engineer Automation Tool (Future)

The JSON schema produced by this app is designed for machine consumption. A future engineer-facing tool will read the exported JSON and automate AD account creation, email provisioning, and application license assignment. The schema uses consistent, unambiguous field names and a stable schemaVersion field to support forward compatibility.
